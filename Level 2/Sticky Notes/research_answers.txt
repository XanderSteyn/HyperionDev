1. HTTP Applications and State Preservation Through the Request-Response Cycle
HTTP is a stateless protocol, which means that each request is treated independently without keeping information from previous interactions. Because of this, web applications have to find ways to maintain user state, mainly by using sessions and authentication. When you log in, the server makes a unique ID for your session. This ID gets saved in a cookie (which is secure and HTTP-only) that the browser sends automatically with each request. Each time you perform an action, your browser sends that ID so that the server knows it's you. That ID links to data stored on the server like your login status or preferences. There are two main ways to handle authentication. Session-based authentication, which means that the server keeps your data and checks the ID every time you send a request. Token-based authentication, like JWTs, which means your information is inside a token that your browser or client app includes in request headers with each request. To optimize performance and scalability, session information might be stored in memory, databases or fast storage solutions like Redis. Cookies help with keeping you logged in or saving settings. On the client side, localStorage or sessionStorage are often used to save non-sensitive data like interface settings, but they're not secure for anything private since they can be accessed through JavaScript. To ensure security, developers set HttpOnly and Secure flags on cookies, which help reduce the risk of Cross-Site Request Forgery (CSRF) attacks (especially when using cookies) and protect tokens from exposure. Even though HTTP forgets everything, these tricks help websites act like they remember you.

2. Django Database Migrations to a Server-Based Relational Database (MariaDB)
First, in the 'settings.py' file update the 'ENGINE' to django.db.backends.mysql. Then, add the host, port, database name, username and password. After that, a driver must be installed. Most people use mysqlclient since it's faster, but PyMySQL also works. Before running migrations, ensure that the target database exists on the MariaDB server. Also, check that the Django user has permissions to create or change tables, otherwise you'll run into errors later. After that, run 'python manage.py makemigrations' to create migration files, and then run 'python manage.py migrate' to apply those changes. It's important to follow best practices, like backing up the database before working on migrations. If a staging environment is available, test there first. Django keeps track of what migrations it's already done in a table called 'django_migrations', so it won't repeat any. If needed, it's possible to see what migrations will do ahead of time by using this command: 'python manage.py migrate --plan'. It's also possible to mark migrations as done without running them if the database is already set up by using this command: 'python manage.py migrate --fake'. For better performance, ensure that indexes have been added where necessary and the correct character sets on MariaDB have been set up.
